# StupidDNS 产品逻辑设计（简化版）

## 设计原则
- **简单优先**：避免过度复杂的逻辑
- **用户无感**：自动化所有决策
- **快速启动**：缓存优先，减少初始化时间

## 核心流程

### 1. 启动初始化

```
启动程序
    ↓
检查配置缓存
    ↓
    ├─ 有缓存且未过期 → 直接加载
    │     ├─ 国内DNS (2个)
    │         └─ 海外DNS (5个)
    │
    └─ 无缓存或已过期 → 执行测速
      ├─ 国内：阿里/腾讯/114 选最快2个
         └─ 海外：预置13个 选最快5个
```

### 2. DNS查询流程

```
收到DNS查询请求
    ↓
检查缓存
    ├─ 命中 → 直接返回
    │
    └─ 未命中
     ↓
    域名分类
    ├─ 中国域名 (china-list) → 使用国内DNS (2个并发)
        │├─ 返回最快结果
    │               └─ 缓存结果
     │
        ├─ GFW域名 (gfwlist) → 使用海外DNS (5个并发)
 │              ├─ 返回最快结果
      │     └─ 缓存结果
        │
        └─ 未知域名 → 双向查询
        ├─ 国内DNS查询 (2个并发)
            ├─ 海外DNS查询 (5个并发)
  ├─ 污染检测
        │   ├─ 国内结果被污染 → 使用海外结果
     │   └─ 国内结果正常 → 比较延迟，选最快
        ├─ 更新动态规则
  └─ 缓存结果
```

### 3. 污染检测规则

```
污染检测(国内DNS结果, 海外DNS结果)
    ├─ 检查是否返回污染IP段
    │   ├─ 8.7.198.45
    │   ├─ 159.106.121.75
    │   ├─ 243.185.187.39
    │   └─ ...（预置污染IP列表）
    │
    ├─ TTL异常检测 (TTL < 100)
    │
    ├─ 多次查询一致性检查
    │
    └─ 判定结果
     ├─ 被污染 → 标记域名为GFW，使用海外结果
      └─ 正常 → 标记域名为国内，使用国内结果
```

## 优化建议

### 简化方案（推荐）
1. **不从网站抓取DNS** - 预置列表足够用
2. **不实现复杂可靠度评分** - 延迟即可靠度
3. **动态规则简化** - 只记录明确的国内/GFW域名
4. **缓存策略简化** - LRU + 24小时过期

### 保持简单的理由
- ? 预置13个海外DNS已经很充分
- ? 用户无感知，自动选择最快的
- ? 减少网络依赖和故障点
- ? 启动速度快
- ? 代码维护简单

## 实现优先级

### P1 (重要但可延后)

7. ? 查询缓存LRU优化
8. ? 统计信息持久化

### P2 (可选)

10. ? 从网站动态获取DNS
11. ? 机器学习域名分类

## 风险点

### 1. 预置DNS失效
**风险**：预置的DNS服务器可能失效
**应对**：
- 定期更新预置列表
- 允许用户手动添加
- 保留多个备选

### 2. 规则文件过时
**风险**：china-list和gfwlist过时
**应对**：
- 提供手动更新功能
- 动态规则补充
- 社区维护规则

### 3. 污染检测误判
**风险**：错误判断域名类型
**应对**：
- 置信度阈值
- 多次验证
- 用户反馈机制

## 性能指标

### 目标
- 缓存命中率: > 80%
- 平均响应时间: < 100ms
- 首次查询: < 300ms
- 启动时间: < 3s（有缓存）
- 内存占用: < 50MB

### 监控
- 总查询数
- 缓存命中数
- 国内/海外查询分布
- 污染检测次数
- 动态规则数量
